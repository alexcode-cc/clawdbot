## 1. 記憶系統概述

OpenClaw 的長期記憶系統是一個**以檔案為真相來源**的架構，使用 Markdown 檔案儲存記憶內容，上層以 SQLite（或 QMD）建立索引來實現語意搜尋。

### 核心原則

- **檔案即真相**：所有記憶儲存為磁碟上的 `.md` 檔案，索引僅為唯讀覆蓋層
- **兩層記憶**：精選的 `MEMORY.md`（長期）+ 日常的 `memory/YYYY-MM-DD.md`（日誌）
- **自動 + 手動**：系統自動沖洗記憶，使用者也可手動編輯
- **語意搜尋**：BM25 關鍵字 + 向量嵌入的混合搜尋

### 記憶來源

| 來源 | 說明 | 預設啟用 |
|------|------|---------|
| `memory` | Markdown 記憶檔案 | 是 |
| `sessions` | JSONL Session 逐字稿 | 否（experimental） |

## 2. 記憶檔案結構與格式

### 目錄結構

```
~/.openclaw/workspace/               # 工作區根目錄
├── MEMORY.md                        # 精選長期記憶（手動維護）
├── memory.md                        # 替代名稱（與 MEMORY.md 共存，去重）
└── memory/                          # 記憶目錄
    ├── 2026-03-20.md                # 每日記憶日誌（自動沖洗寫入）
    ├── 2026-03-21.md
    ├── 2026-03-22.md
    ├── 2026-03-22-api-design.md     # Session 記憶（Hook 自動建立）
    ├── 2026-03-23-bug-investigation.md
    └── topics.md                    # 主題分類記憶（手動建立）

~/.openclaw/agents/<agentId>/        # Agent 專屬目錄
├── <agentId>.sqlite                 # 記憶索引資料庫
└── sessions/                        # Session 逐字稿
    ├── session-abc123.jsonl
    └── session-def456.jsonl
```

### MEMORY.md — 精選長期記憶

**用途**：手動維護的精選記憶，包含重要的長期參考資訊。

**特性**：
- 在主要私人 session 中於啟動時載入
- 群組/頻道 context 中不載入
- Agent 記憶沖洗時**不會**修改此檔案（唯讀保護）
- 可由使用者或 Agent（透過明確指令）手動編輯

**建議內容**：
```markdown
# Long-term Memory

## 使用者偏好
- 使用繁體中文回覆
- 偏好簡潔的技術回覆
- 工作環境：Mac Mini M4 + Ubuntu Server

## 專案筆記
- 主要專案：OpenClaw 中文文件翻譯
- Git 分支策略：feature → develop → main

## 重要聯絡人
- @admin_user — 主要管理員
- @dev_team — 開發團隊頻道
```

### memory/YYYY-MM-DD.md — 每日記憶日誌

**用途**：自動記憶沖洗的目標檔案，追加式寫入。

**命名規則**：使用使用者設定的時區產生日期（如 `Asia/Taipei`）。

**特性**：
- **僅追加（Append-only）**：新內容追加至檔尾，不覆寫現有內容
- Session 啟動時載入今天 + 昨天的日誌
- 自動沖洗時由 Agent 決定寫入內容

**典型內容**：
```markdown
## 10:30 — API 設計討論

與 @dev 討論了新的 REST API 設計：
- 採用 /v2/resources 路徑
- 認證改用 Bearer token
- 分頁使用 cursor-based pagination

## 14:15 — 部署問題排查

Docker 容器啟動失敗，原因是 Node.js 版本不匹配：
- 將 Dockerfile 從 node:20 升級到 node:22
- 重新建置後問題解決

## 17:00 — 程式碼審查

審查了 PR #456：
- 修正了 memory flush 的追加邏輯
- 新增了 3 個單元測試
```

### memory/YYYY-MM-DD-slug.md — Session 記憶

**用途**：由 `session-memory` Hook 在 `/new` 或 `/reset` 時自動建立。

**命名規則**：`YYYY-MM-DD-{LLM 生成的 slug}.md`

**典型內容**：
```markdown
# Session: 2026-03-23 14:30:00 UTC

- **Session Key**: agent:main:telegram:direct:123456
- **Session ID**: abc123def456
- **Source**: telegram

## Conversation Summary

User: 幫我分析這個 API 的效能問題
Assistant: 分析了 /api/search 端點，發現 N+1 查詢問題...
User: 如何修復？
Assistant: 建議使用 DataLoader 批次查詢...
[... 最後 15 則訊息 ...]
```

## 3. 記憶的建立方式

OpenClaw 記憶可透過四種方式建立：

### 3.1 自動記憶沖洗（Memory Flush）

- **觸發時機**：Session 接近 context window 上限時，compaction 前自動執行
- **目標檔案**：`memory/YYYY-MM-DD.md`（追加）
- **行為**：Agent 靜默地將「持久性」資訊寫入記憶檔案
- 詳見 [第 4 節](#4-自動記憶沖洗memory-flush)

### 3.2 Session-Memory Hook

- **觸發時機**：使用者執行 `/new` 或 `/reset` 時
- **目標檔案**：`memory/YYYY-MM-DD-slug.md`（新建）
- **行為**：自動擷取前一 session 的最後 N 則訊息
- 詳見 [第 5 節](#5-session-memory-hook)

### 3.3 Agent 主動寫入

Agent 可透過 `write` 工具在對話中主動寫入記憶：

```
User: 記住我的偏好：回覆時使用繁體中文
Agent: [使用 write 工具寫入 memory/2026-03-23.md]
```

### 3.4 使用者手動編輯

使用者可直接編輯工作區中的 `.md` 檔案：
- 編輯 `MEMORY.md` 管理長期精選記憶
- 在 `memory/` 目錄新增或修改任何 `.md` 檔案
- 檔案監視器（watcher）會自動偵測變更並重新索引

## 4. 自動記憶沖洗（Memory Flush）

### 觸發條件

記憶沖洗在 **compaction（上下文壓縮）前** 自動觸發，條件為：

| 條件 | 預設值 | 說明 |
|------|--------|------|
| Token 軟閾值 | `contextWindow - reserveTokensFloor - 4000` | Session token 接近上限 |
| Transcript 大小 | 2 MB | Session 逐字稿超過 2 MB 強制觸發 |

**排除條件**（以下情況不觸發）：
- 工作區為唯讀（`ro`）或無存取（`none`）沙箱
- CLI provider run
- 心跳（heartbeat）session
- 同一 compaction 週期內已沖洗過

### 沖洗流程

```
1. 偵測到即將 compaction
        │
        ▼
2. 計算 context hash（最後 3 則訊息的 SHA-256）
   ├── 與前次相同 → 跳過（避免重複沖洗）
   └── 不同 → 繼續
        │
        ▼
3. 建構沖洗 prompt（含安全提示）
        │
        ▼
4. 以受限工具集執行 Agent turn
   ├── 僅允許 read + write 工具
   └── write 工具加上追加保護 wrapper
        │
        ▼
5. Agent 決定要記住什麼，寫入 memory/YYYY-MM-DD.md
        │
        ▼
6. Agent 回覆 NO_REPLY（靜默 turn）
        │
        ▼
7. 標記此 compaction 週期已沖洗
```

### 沖洗時使用的 Prompt

**系統 Prompt**：
```
Pre-compaction memory flush turn.
The session is near auto-compaction; capture durable memories to disk.
Store durable memories only in memory/YYYY-MM-DD.md (create memory/ if needed).
Treat workspace bootstrap/reference files such as MEMORY.md, SOUL.md, TOOLS.md,
and AGENTS.md as read-only during this flush; never overwrite, replace, or edit them.
If memory/YYYY-MM-DD.md already exists, APPEND new content only and do not
overwrite existing entries.
You may reply, but usually NO_REPLY is correct.
```

**使用者 Prompt**：
```
Pre-compaction memory flush.
Store durable memories only in memory/YYYY-MM-DD.md (create memory/ if needed).
Treat workspace bootstrap/reference files such as MEMORY.md, SOUL.md, TOOLS.md,
and AGENTS.md as read-only during this flush; never overwrite, replace, or edit them.
If memory/YYYY-MM-DD.md already exists, APPEND new content only and do not
overwrite existing entries.
Do NOT create timestamped variant files (e.g., YYYY-MM-DD-HHMM.md);
always use the canonical YYYY-MM-DD.md filename.
If nothing to store, reply with NO_REPLY.
```

### 安全保護機制

| 保護 | 說明 |
|------|------|
| **目標檔案限定** | 僅寫入 `memory/YYYY-MM-DD.md` |
| **Bootstrap 唯讀** | MEMORY.md、SOUL.md、TOOLS.md、AGENTS.md 不可修改 |
| **追加模式** | 已存在的日誌僅追加，不覆寫 |
| **工具限制** | 僅 `read` 和 `write` 工具可用 |
| **Write wrapper** | `write` 工具加上強制追加保護 |
| **無變體檔案** | 禁止建立 `YYYY-MM-DD-HHMM.md` 變體 |
| **去重** | 同一 compaction 週期不重複沖洗 |
| **Context hash** | 內容無變化時跳過沖洗 |

### 沖洗設定

```json5
{
  agents: {
    defaults: {
      compaction: {
        reserveTokensFloor: 20000,         // 預留 token 下限
        memoryFlush: {
          enabled: true,                   // 啟用自動沖洗
          softThresholdTokens: 4000,       // 軟閾值（tokens）
          forceFlushTranscriptBytes: 2097152, // 強制閾值（2 MB）
          // 自訂 prompt（需包含三項安全提示，否則自動附加）
          // systemPrompt: "...",
          // prompt: "...",
        },
      },
    },
  },
}
```

## 5. Session-Memory Hook

### 概述

`session-memory` 是一個 bundled hook，在 `/new` 或 `/reset` 時自動將前一 session 的對話摘要儲存為記憶檔案。

### 運作流程

```
1. 使用者執行 /new 或 /reset
        │
        ▼
2. Hook 觸發，讀取前一 session 的 JSONL 逐字稿
        │
        ▼
3. 擷取最後 N 則 user/assistant 訊息（預設 15 則）
        │
        ▼
4. 呼叫 LLM 從對話內容生成描述性 slug
   （失敗時 fallback 為時間戳 HHMM）
        │
        ▼
5. 建立檔案：memory/YYYY-MM-DD-{slug}.md
   （包含 session metadata + 對話內容）
```

### 設定

```json5
{
  hooks: {
    internal: {
      enabled: true,
      entries: {
        "session-memory": {
          enabled: true,
          messages: 15,           // 保留的訊息數（預設 15）
          llmSlug: true,          // 使用 LLM 生成檔名（預設 true）
        },
      },
    },
  },
}
```

### 啟用方式

```bash
# 透過 CLI 啟用
openclaw hooks enable session-memory

# 或在 onboard 精靈的 Hooks 步驟中選擇啟用
openclaw onboard
```

## 6. 記憶索引與搜尋系統

### 索引架構

```
Markdown 檔案 (.md)
      │
      ▼
  分塊 (Chunking)
  ├── 每塊 400 tokens
  └── 80 tokens 重疊
      │
      ▼
  嵌入向量 (Embedding)          全文索引 (FTS5)
  ├── Provider API 呼叫         ├── SQLite FTS5
  └── 儲存至 chunks_vec          └── 儲存至 chunks_fts
      │                              │
      └────────┬─────────────────────┘
               │
               ▼
         混合搜尋 (Hybrid Search)
         ├── 向量相似度 × 0.7
         ├── BM25 關鍵字 × 0.3
         ├── MMR 多樣性重排序（可選）
         └── 時間衰減（可選）
               │
               ▼
         排序結果 → Agent
```

### 索引資料庫

- **驅動**：SQLite（預設）
- **位置**：`~/.openclaw/agents/<agentId>.sqlite`（可設定）
- **表結構**：
  - `meta` — key-value 設定儲存
  - `files` — 已索引檔案（path、hash、mtime、size）
  - `chunks` — 文字分塊（含嵌入向量）
  - `chunks_vec` — 向量搜尋表（sqlite-vec 加速）
  - `chunks_fts` — 全文搜尋虛擬表（FTS5）
  - `embedding_cache` — 嵌入快取（按 provider/model/hash）

### 索引同步

| 觸發時機 | 設定 | 預設 |
|---------|------|------|
| Session 啟動時 | `sync.onSessionStart` | `true` |
| 搜尋時 | `sync.onSearch` | `true` |
| 檔案變更時 | `sync.watch` | `true`（debounce 1.5s） |
| 定期同步 | `sync.intervalMinutes` | `0`（停用） |

## 7. 嵌入向量提供者

### 支援的提供者

| 提供者 | 預設模型 | 維度 | 需要 API Key | Auto 偵測順序 |
|--------|---------|------|-------------|-------------|
| `local` | embeddinggemma-300m-qat-Q8_0 | — | 否 | 1（需設定 modelPath） |
| `openai` | text-embedding-3-small | 1536 | 是 | 2 |
| `gemini` | gemini-embedding-001 | 768/1536/3072 | 是 | 3 |
| `voyage` | voyage-4-large | — | 是 | 4 |
| `mistral` | mistral-embed | — | 是 | 5 |
| `ollama` | nomic-embed-text | — | 否 | 不自動選擇 |

### Auto 模式偵測

當 `provider: "auto"` 時，依序嘗試：
1. `local`（需 `memorySearch.local.modelPath` 已設定且檔案存在）
2. `openai`（需 OpenAI API Key 可用）
3. `gemini`（需 Gemini API Key 可用）
4. `voyage`（需 Voyage API Key 可用）
5. `mistral`（需 Mistral API Key 可用）
6. 全部不可用 → 記憶搜尋停用

### 提供者設定範例

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        // 使用 OpenAI
        provider: "openai",
        model: "text-embedding-3-small",
        remote: {
          apiKey: "${OPENAI_API_KEY}",
        },

        // 或使用 Ollama（本地）
        // provider: "ollama",
        // model: "nomic-embed-text",
        // remote: {
        //   baseUrl: "http://localhost:11434",
        // },

        // 或使用本地模型（完全離線）
        // provider: "local",
        // local: {
        //   modelPath: "~/.openclaw/models/embeddinggemma-300m.gguf",
        //   modelCacheDir: "~/.openclaw/cache/models",
        // },

        // Fallback 提供者（主要不可用時使用）
        fallback: "local",              // 或 "none" 停用 fallback
      },
    },
  },
}
```

### 多模態嵌入

僅 Gemini embedding-2 支援圖片和音訊的嵌入：

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "gemini",
        model: "gemini-embedding-2-preview",
        multimodal: {
          enabled: true,
          modalities: ["image", "audio"],    // 或 ["all"]
          maxFileBytes: 10485760,            // 10 MB
        },
        // 額外路徑（多模態檔案僅從 extraPaths 載入）
        extraPaths: ["~/photos/reference/"],
      },
    },
  },
}
```

**支援的多模態格式**：
- 圖片：`.jpg`、`.jpeg`、`.png`、`.webp`、`.gif`、`.heic`、`.heif`
- 音訊：`.mp3`、`.wav`、`.ogg`、`.opus`、`.m4a`、`.aac`、`.flac`

## 8. 混合搜尋（Hybrid Search）

### 搜尋流程

```
查詢文字
    │
    ├──────────────────┐
    ▼                  ▼
向量搜尋              BM25 關鍵字搜尋
(K × candidateMultiplier)    (K 筆結果)
    │                  │
    └────────┬─────────┘
             ▼
    合併（by chunk ID）
    加權計分：vector×0.7 + text×0.3
             │
             ▼
    MMR 多樣性重排序（可選）
    λ × relevance - (1-λ) × max_similarity
             │
             ▼
    時間衰減（可選）
    score × exp(-λ × ageInDays)
             │
             ▼
    返回 Top K 結果
```

### 搜尋設定

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        query: {
          maxResults: 6,                    // 返回筆數
          minScore: 0.35,                   // 最低分數門檻

          hybrid: {
            enabled: true,                  // 啟用混合搜尋
            vectorWeight: 0.7,              // 向量權重（正規化後）
            textWeight: 0.3,                // 文字權重（正規化後）
            candidateMultiplier: 4,         // 候選倍數（重排序用）

            // MMR（最大邊際相關性）
            mmr: {
              enabled: false,               // 啟用多樣性重排序
              lambda: 0.7,                  // 0=最大多樣性，1=最大相關性
            },

            // 時間衰減
            temporalDecay: {
              enabled: false,               // 啟用時間衰減
              halfLifeDays: 30,             // 半衰期（天）
            },
          },
        },
      },
    },
  },
}
```

### 時間衰減規則

- **日期化檔案**（`memory/2026-03-20.md`）：依檔名日期計算年齡
- **常青檔案**（`MEMORY.md`、`memory/topics.md`）：**永不衰減**
- **衰減公式**：`score × exp(-ln(2)/halfLifeDays × ageInDays)`
- **效果**：halfLifeDays=30 時，30 天前的記憶分數減半

### 多語言支援

搜尋系統支援多語言的停用詞過濾和查詢擴展：

| 語言 | 支援 |
|------|------|
| 英語 | 完整停用詞列表 |
| 西班牙語 | 停用詞 |
| 葡萄牙語 | 停用詞 |
| 阿拉伯語 | 停用詞 |
| 韓語 | 助詞剝離 |
| 日語 | 特殊 N-gram 處理 |
| 中文 | 特殊 N-gram 處理 |

## 9. Agent 記憶工具

### memory_search — 語意搜尋

**功能**：對已索引的記憶進行語意搜尋。

**參數**：
```typescript
{
  query: string;          // 必要：搜尋查詢
  maxResults?: number;    // 可選：結果數量上限
  minScore?: number;      // 可選：最低分數門檻
}
```

**回傳結果**：
```typescript
{
  results: [
    {
      snippet: string;     // 約 700 字元摘要（含引用）
      path: string;        // 相對路徑（如 "memory/2026-03-23.md"）
      startLine: number;   // 起始行（1-indexed）
      endLine: number;     // 結束行
      score: number;       // 相關性分數（0-1）
      citation: string;    // "path#L123-L456"（citations 啟用時）
    },
    // ...
  ],
  provider: string;        // 使用的嵌入提供者
  model: string;           // 使用的嵌入模型
}
```

### memory_get — 目標讀取

**功能**：讀取特定記憶檔案的內容。

**參數**：
```typescript
{
  path: string;            // 必要：檔案路徑（如 "memory/2026-03-23.md"）
  from?: number;           // 可選：起始行（1-indexed）
  lines?: number;          // 可選：讀取行數
}
```

**回傳**：檔案內容（檔案不存在時回傳空字串，不拋錯）。

### 引用模式

| 模式 | 行為 |
|------|------|
| `auto`（預設） | 私人 DM 中顯示引用，群組/頻道中隱藏 |
| `on` | 總是顯示引用 |
| `off` | 總是隱藏引用（但 Agent 仍可用 path 呼叫 memory_get） |

## 10. QMD 進階後端

### 概述

QMD 是一個**實驗性**的外部側車二進位檔，提供 BM25 + 向量 + 重排序功能。Markdown 仍為真相來源，QMD 僅作為檢索引擎。

### 基本設定

```json5
{
  memory: {
    backend: "qmd",                        // "builtin"（預設）或 "qmd"
    qmd: {
      command: "qmd",                      // QMD 二進位路徑
      searchMode: "search",               // "search"（快速）| "vsearch"（向量）| "query"（慢，含擴展+重排序）
      includeDefaultMemory: true,          // 包含預設記憶目錄

      // 自訂索引路徑
      paths: [
        { path: "~/docs/notes", pattern: "**/*.md", name: "notes" },
        { path: "/srv/shared", name: "shared" },
      ],

      // Session 匯出
      sessions: {
        enabled: false,
        exportDir: "~/.openclaw/agents/{agentId}/qmd/sessions",
        retentionDays: 30,
      },

      // 更新排程
      update: {
        interval: "5m",                    // 更新間隔
        debounceMs: 15000,                 // 變更 debounce
        onBoot: true,                      // 啟動時同步
        waitForBootSync: false,            // 等待啟動同步完成
        embedInterval: "60m",              // 嵌入更新間隔
      },

      // 搜尋限制
      limits: {
        maxResults: 6,
        maxSnippetChars: 700,
        maxInjectedChars: 4000,
        timeoutMs: 4000,
      },
    },
  },
}
```

> QMD 失敗時自動 fallback 至內建 SQLite 後端。

## 12. 記憶濃縮與清除策略

### 清除舊記憶

#### 刪除特定日期的記憶

```bash
# 刪除單一日期
rm ~/.openclaw/workspace/memory/2026-01-15.md

# 刪除一個月前的所有記憶
find ~/.openclaw/workspace/memory/ -name "2026-01-*.md" -delete

# 刪除所有 Session 記憶（保留每日日誌）
find ~/.openclaw/workspace/memory/ -name "????-??-??-*.md" -delete

# 保留最近 30 天，刪除更舊的
find ~/.openclaw/workspace/memory/ -name "????-??-??.md" -mtime +30 -delete
```

#### 清空並重建索引

```bash
# 刪除索引資料庫（下次搜尋時自動重建）
rm ~/.openclaw/agents/main/main.sqlite

# 或強制重新索引
openclaw memory index --force
```

### 濃縮（Consolidation）策略

當記憶檔案過多或過大時，建議以下濃縮策略：

#### 策略 1：手動精選至 MEMORY.md

1. 瀏覽近期的 `memory/YYYY-MM-DD.md` 檔案
2. 將重要的長期資訊提取至 `MEMORY.md`
3. 刪除已整合的每日記憶

```bash
# 查看近期記憶
for f in ~/.openclaw/workspace/memory/2026-03-*.md; do
  echo "=== $(basename $f) ==="
  head -20 "$f"
  echo ""
done

# 編輯精選記憶，加入重要內容
vim ~/.openclaw/workspace/MEMORY.md

# 刪除已整合的來源
rm ~/.openclaw/workspace/memory/2026-03-{01..15}.md
```

#### 策略 2：按主題分類整理

在 `memory/` 目錄下建立主題檔案：

```
memory/
├── topics.md              # 主題索引
├── projects.md            # 專案筆記
├── preferences.md         # 偏好設定
├── contacts.md            # 聯絡人資訊
├── 2026-03-23.md          # 近期每日日誌
└── 2026-03-24.md
```

> 主題檔案（非日期命名）不受時間衰減影響。

#### 策略 3：請 Agent 協助濃縮

在對話中指示 Agent 進行濃縮：

```
User: 請閱讀 memory/ 目錄下 3 月 1 日到 15 日的所有記憶檔案，
將重要資訊整理成摘要，寫入 memory/2026-03-summary.md，
然後刪除原始的每日檔案。
```

#### 策略 4：啟用時間衰減

讓搜尋系統自動降低舊記憶的權重：

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        query: {
          hybrid: {
            temporalDecay: {
              enabled: true,
              halfLifeDays: 30,    // 30 天半衰期
            },
          },
        },
      },
    },
  },
}
```

這不會刪除舊記憶，但搜尋時會優先返回近期記憶。

### 記憶大小監控

```bash
# 查看記憶目錄大小
du -sh ~/.openclaw/workspace/memory/

# 查看索引資料庫大小
du -sh ~/.openclaw/agents/main/main.sqlite

# 查看檔案數量
ls ~/.openclaw/workspace/memory/ | wc -l

# 按大小排序記憶檔案
ls -lhS ~/.openclaw/workspace/memory/
```

### 自動清除腳本範例

建立一個定期清除的 cron 腳本：

```bash
#!/bin/bash
# cleanup-memory.sh - 保留最近 60 天的每日記憶

MEMORY_DIR="$HOME/.openclaw/workspace/memory"
DAYS_TO_KEEP=60

# 找到並刪除超過 60 天的每日記憶（YYYY-MM-DD.md 格式）
find "$MEMORY_DIR" -maxdepth 1 -name "????-??-??.md" -mtime +$DAYS_TO_KEEP -delete

# 找到並刪除超過 90 天的 Session 記憶（YYYY-MM-DD-slug.md 格式）
find "$MEMORY_DIR" -maxdepth 1 -name "????-??-??-*.md" -mtime +90 -delete

echo "Memory cleanup completed: $(date)"
```

搭配 cron 每週執行：
```bash
# crontab -e
0 3 * * 0 /path/to/cleanup-memory.sh >> /tmp/memory-cleanup.log 2>&1
```

## 15. 注意事項與限制

### 重要須知

1. **記憶非揮發性**：所有記憶必須寫入磁碟才能持久化，RAM 中的資訊在 session 結束後遺失
2. **分塊以 token 計**：搜尋結果的行號對應原始檔案行號，非分塊邊界
3. **Session 索引需 opt-in**：`experimental.sessionMemory = true` 才會索引 session 逐字稿
4. **多模態僅限 Gemini**：僅 `gemini-embedding-2-preview` 支援圖片/音訊嵌入
5. **QMD 完全可選**：內建 SQLite 已可生產使用，QMD 為實驗性替代方案
6. **sqlite-vec 為可選加速**：不可用時 fallback 至記憶體內餘弦相似度
7. **FTS5 依賴 SQLite 編譯選項**：不可用時關鍵字搜尋優雅降級
8. **沖洗每 compaction 週期最多一次**：防止重複沖洗
9. **檔案監視 debounce**：快速連續編輯在 1.5 秒內合併為一次重新索引
10. **嵌入需要 API Key**：遠端提供者需要對應的 API Key，本地提供者不需要

### 效能考量

| 項目 | 建議 |
|------|------|
| 記憶檔案數量 | 數百個檔案可正常運作，數千個建議定期清理 |
| 單檔大小 | 建議 < 1 MB（索引上限依設定而定） |
| 索引資料庫 | 通常 < 100 MB，可安全刪除重建 |
| 嵌入 API 呼叫 | 首次索引可能較慢，後續增量快取 |
| 搜尋延遲 | 通常 < 1 秒（sqlite-vec 加速時） |

### 常見問題

| 問題 | 解決方案 |
|------|---------|
| 記憶搜尋無結果 | 確認 `memorySearch.enabled: true`，檢查 `openclaw memory status --deep` |
| 嵌入失敗 | 確認 API Key 有效，執行 `openclaw memory status --deep` |
| 索引過時 | 執行 `openclaw memory index --force` |
| 記憶檔案過多 | 見 [第 12 節](#12-記憶濃縮與清除策略) 清除策略 |
| Session 記憶未建立 | 確認 `session-memory` hook 已啟用 |
| 沖洗未觸發 | 確認 `compaction.memoryFlush.enabled: true` |
| 沖洗覆寫了記憶 | 確認使用最新版本（追加保護已內建，2026.3.28 修復） |
| CJK 搜尋結果不佳 | 確認使用 2026.3.28+（CJK chunking 和 MMR tokenize 已修復） |

### CJK 記憶支援改善（2026.3.28+）

自 2026.3.28 起，記憶系統對中日韓語言有以下重大改善：

1. **QMD 分塊**：正確計算 CJK 字元（中/日/韓/越南語漢字）的 token 數量，避免分塊過大或過小
2. **UTF-16 代理對保護**：fine-split 不再切斷 UTF-16 surrogate pairs（Emoji 等擴充字元安全）
3. **MMR 多樣性**：`tokenize()` 正確分詞 CJK/Kana/Hangul 字元，多樣性重排序不再偏好拉丁語系
4. **Memory Flush 追加修復**：修復 2026.3.24-3.27 版本中可能覆寫（而非追加）每日記憶檔案的回歸
